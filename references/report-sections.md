# 主报告章节语义说明

这个文件解释主报告各章节分别负责什么，避免在 `SKILL.md` 里反复堆叠说明。

## 1. 项目在解决什么问题

回答：

- 这个项目面向什么目标
- 它服务什么用户或工程对象
- 它试图替代、补足或组织什么东西

不要把 README 口号直接改写成结论。

## 2. 核心部件地图

回答：

- 系统里有哪些关键部件
- 它们分别承担什么职责
- 它们如何协作

这一节偏“横向地图”，不负责解释目录分层细节。

## 3. 项目结构设计

回答：

1. 仓库或模块是怎么分层的
2. 每层各自负责什么
3. 边界是怎么切的
4. 这种结构和主链路之间是什么关系

这一节优先解释：

- 目录结构
- 包结构
- 分层责任
- 适配层、编排层、领域层、接口层等边界

不要只重复“有这些文件”。要回答“为什么这样分”和“这样分以后主链路怎么走”。

## 4. 一次典型流程怎么跑完

回答：

- 从入口到结果的一条主链路
- 各阶段由谁负责
- 关键状态或数据怎么流动

这节偏“纵向过程”。

## 5. 关键代码导读

回答：

- 哪些代码段最能支撑前面的结构和流程判断
- 这些代码在系统里负责什么

不要只堆路径和片段。

## 6. 技术栈与关键依赖

这一节保持轻量，只回答：

1. 明确能确认的语言、框架、运行时、存储、协议、构建工具
2. 它们分别承担什么作用
3. 哪些是直接证据，哪些只是推断

不要把它写成依赖清单，也不要让它抢走结构设计和主链路的篇幅。

## 7. Why：为什么它可能这样设计

回答：

- 为什么会选这种结构
- 为什么主链路会这样组织
- 设计权衡可能落在哪

这里允许工程推断，但要显式标出判断口气。

## 8. 证据分层

固定分成：

- 直接证据
- 工程推断
- 外部支持

## 9. 可行性评估

回答：

- 这套设计在什么前提下成立
- 适合什么场景
- 不适合什么场景

## 10. 风险、未知项与边界

回答：

- 当前还不能确认什么
- 潜在失败模式在哪
- 结论依赖哪些样本边界

## 11. 最终判断

给出收束后的整体结论，不再引入大段新信息。
