# Evidence-Driven Analysis

这个 skill 用来产出高强度、证据分层明确、但同时对新人友好的技术分析与架构讲解文档。

默认规则：跟随使用者当前语言输出。若用户使用中文，则分析过程、解释、结论、模板文本优先使用中文；外部资料标题、接口名、项目名、字段名可以保留原文。这是内部执行规则，不要把“本文遵守……原则”“以下以中文撰写”之类的写作元说明直接写进最终产物。

## 核心原则

1. 先帮助读者建立全局，再进入细节。
2. 先收集直接证据，再做工程推断。
3. 当前且易变化的信息必须核验，不要凭记忆硬答。
4. 严格区分“直接证据”“工程推断”“外部支持”。
5. 不要把 README 口径直接当成实现事实。
6. 输出应服务于使用者语言，而不是服务于资料原文语言。

## 默认产物风格

当用户是在理解项目、系统或架构，而不是只做研究判断时，默认把产物写成“由浅入深的架构导览”，不是纯研究报告。

优先采用这个顺序：

1. 读法说明
2. 总览图或结构图
3. 5 分钟认识系统
4. 核心部件地图
5. 一次典型流程怎么跑完
6. 关键代码导读
7. 再进入 why / feasibility / risk / comparison

## 工作流

1. 界定目标对象、读者、范围、成功标准。
2. 判断产物是“研究判断型”还是“新人导览型”；若用户要普及、 onboarding、讲架构，默认走新人导览型。
3. 建立证据账本：直接证据、工程推断、外部支持。
4. 重建 `what / how / why`。
5. 如果适合图示，优先给出总览图、流程图、部件表，而不是长段抽象判断。
6. 在关键链路处提供“代码片段 + 中文解释”并排讲解。
7. 必要时对照主流实践、官方文档、论文或成熟项目。
8. 默认把结果落成一个本地 Markdown 文档，而不是只停留在聊天回复里；如果已经存在同一研究对象的文档集，后续追问也必须继续落盘，不能只在聊天里补几段文字。
9. 第一次成稿时，创建主报告（canonical report）。若用户没有指定文件名，默认使用 `<project-name>-report.md`；如果项目名暂时无法可靠判断，再退回 `architecture-report.md`。
10. 后续追问要先判断：这是继续补主报告，还是已经形成值得独立成篇的子专题。这个 skill 对独立专题采取偏积极拆分策略，但不会因为任何后续问题都新建文档。
11. 如果后续问题已经可独立命名、可独立阅读、需要独立证据链，优先新建或更新子文档，而不是把所有内容都塞回主报告。
12. 如果子问题产出了会改变全局理解的结论，要把精华版结论回灌到主报告的合适位置，并保留对子文档的引用。
13. 最终回复要同时给出本轮更新了哪些文档、哪个是主报告、哪个是子文档，并用一两句话概括新增覆盖的重点。

## 多轮文档演进规则

### 主报告

- 第一次针对某个项目、系统、论文或方案形成实质性分析时，创建一份主报告。
- 主报告是该研究对象的稳定入口，负责承载总览、主链路、关键部件地图、整体判断和后续阅读入口。
- 后续即使拆出了多个子文档，主报告仍然应该保持“先读这个就能建立全局”的作用。

### 何时优先拆成子文档

这个 skill 对独立专题采取偏积极拆分策略，但不是“后面只要继续问就新建文档”。只有当一个后续问题同时满足下面三个条件时，才优先新建或更新子文档：

1. 这个问题可以被清楚命名成一个独立专题。
2. 回答它需要一段相对独立的证据链、结构化展开或完整小结论。
3. 如果把这轮内容直接塞回主报告，会明显打断主线，或让主报告结构失衡。

在满足上面三个条件的前提下，下面这些信号会进一步支持拆分：

1. 用户开始深挖单一模块、单一流程、单一风险、单一对比维度。
2. 研究视角发生切换，例如从架构导览切到协议设计、性能、迁移、可行性、失败模式、边界条件。
3. 用户在语气或提问结构上，已经把它当作一个值得单独展开的专题。

### 何时继续写主报告

只有在下面这些情况，才优先直接更新主报告：

1. 这轮只是补几条新证据。
2. 这轮只是把现有章节讲清楚一点。
3. 这轮只是补一张图、一个代码导读块，或一个小范围 caveat。
4. 这轮问题还不足以形成独立专题名，或者不需要独立证据链就能说清。
5. 这轮问题只是换了一种问法追问原主题，没有改变主报告的结构重心。

### 不要过度拆分

下面这些情况，默认不应新建子文档：

1. 用户只是追问一句补充解释。
2. 用户只要一个例子、一段额外证据、一个更清楚的表述。
3. 问题虽然换了说法，但本质上仍在原章节的自然延长线上。
4. 新内容完全可以作为主报告中的一个小节或几段补充来承载。

### 子文档不是孤岛

- 子文档负责展开独立专题，不负责取代主报告。
- 如果子文档得出了会影响全局理解的新结论、边界、风险或设计解释，要把提炼版内容回灌到主报告的对应位置。
- 回灌的是结论、边界、风险、链接和一两句必要解释，不是整段复制。
- 如果子文档没有改变主报告的主结论，也要在主报告里保留入口，让读者知道这个专题已经被单独展开。

### 命名约定

- 主报告：`<project-name>-report.md`
- 独立追问型子文档：`<project-name>-<topic>-followup.md`
- 材料补充型子文档：`<project-name>-<topic>-appendix.md`

`followup` 用于可以独立阅读的子问题分析。
`appendix` 用于证据整理、专项代码导读、矩阵、对照表等偏材料型补充。

### 互链约定

- 新建子文档时，开头就说明：它聚焦什么子问题、与主报告是什么关系、主报告文件是什么。
- 主报告应维护一个稳定的 `## 后续专题 / 继续阅读` 区域。
- 每新增一个子文档，都在这个区域添加一句话摘要和文件引用。
- 如果这轮显著更新了既有子文档，必要时同步更新主报告里的摘要。

### 每轮落盘契约

后续追问不能只停留在聊天里。本轮至少完成下面一项：

1. 更新主报告
2. 新建或更新子文档
3. 同时做两者

最终聊天回复应明确说明：

- 这次更新了主报告、子文档，还是两者都更新了
- 相关文件路径或文件名
- 本轮关键结论是否已经回灌到主报告

## 图文并茂规则

- 只要主题涉及架构、运行链路、模块协作，就优先给图。
- 如果不能安全或方便生成图片，至少给 Mermaid 图、ASCII 图、表格或编号流程。
- 图的作用是先帮读者抓整体，不是装饰。
- 图里的关键标签优先使用读者语言；如果必须保留英文术语，就写成 `AgentLoop（主循环）` 这种形式。
- Mermaid 优先选兼容性更高的简单写法，例如 `flowchart TD/LR`；如果目标阅读器容易把图渲染坏，就退回 ASCII、表格或编号流程。
- 在 Markdown 阅读器里，总览图如果横向太长、字太密，优先改成纵向布局，或拆成两张更易读的图。
- 代码块不要孤立摆放；每个代码块后面都要回答“这段代码在系统里负责什么”。

## 表达约束

- 少用“不是 A 而是 B”“换言之”“这意味着”这一类空转句式。
- 优先写事实句、动作句、对比句。
- 当需要做比较，写清楚“比较维度”是什么，例如“结构约束放在哪一层”“会话状态放在哪一层”。
- 当读者是新人，先解释名词在系统里起什么作用，再解释它的理论归类。
- 项目内的关键术语第一次出现时，优先写成“英文名或原名（中文作用解释）”；后文再按上下文决定是否用简称。
- 不要在成稿里出现“本文遵守某原则”“以下内容使用某语言”“为了方便理解”这类写作元说明；直接把规则落实到正文风格里。

## 何时读取 references

- 需要定义证据强度时，读取 `references/evidence-model.md`
- 需要系统评估可行性与风险时，读取 `references/feasibility-checklist.md`
- 需要输出新人友好的完整成稿时，读取 `references/report-template.md`
- 需要参考本仓库示例时，读取 `references/senera-loop-example.md`

## 输出契约

默认输出应包含：

1. 分析对象与读者边界
2. 总览图或结构图
3. `what / how / why`
4. 关键流程讲解
5. 代码与解释并行的导读块
6. 证据分层
7. 可行性判断
8. 风险与未知项
9. 最终结论
10. 一个已经写入本地的 Markdown 报告文件；除非用户明确要求只在对话中回答，否则不要只回聊天文本

如果这是同一研究对象的后续轮次，还应包含：

11. 说明本轮是更新主报告、更新子文档，还是同时更新两者
12. 如果新建了子文档，主报告中的 `## 后续专题 / 继续阅读` 应同步出现入口
13. 如果子文档产出了影响全局理解的新结论，主报告对应章节应吸收其精华版结论

## 执行提醒

- 先用本地材料回答“代码里到底有什么”，再用外部材料回答“这种设计在公共世界里站不站得住”。
- 解释性句子要显式带上判断口气，例如“可以推断”“更偏向”“这里看起来是在解决……”，不要把推断伪装成事实。
- 外部来源除了说明“支持了什么”，还要说明“没有证明什么”，尤其不要把厂商文档或论文写成对当前实现的直接验收。
- 当样本有限时，把判断收窄到“从本次引用资料看”或“在当前采样到的公开实践中”，不要擅自外推成整个行业共识。
- 当用户要的是新人可读性，先讲“系统怎么跑”，再讲“系统为什么这样设计”。
- 如果已经落盘成文档，聊天回复里优先报告文件路径和摘要，不要把整份长文再原样重复一遍。
