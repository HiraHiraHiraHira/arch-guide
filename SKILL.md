# Evidence-Driven Analysis

这个 skill 用来产出证据分层明确、对新人友好、适合落成文档的技术分析。

默认规则：跟随使用者当前语言输出。若用户使用中文，则分析过程、解释、结论、模板文本优先使用中文；外部资料标题、接口名、项目名、字段名可以保留原文。不要把这些语言规则写成正文元说明。

## 核心原则

1. 先帮助读者建立全局，再进入细节。
2. 先收集直接证据，再做工程推断。
3. 当前且易变化的信息必须核验，不要凭记忆硬答。
4. 严格区分“直接证据”“工程推断”“外部支持”。
5. 不要把 README 口径直接当成实现事实。
6. 输出服务于读者理解，不服务于资料原文顺序。

## 何时使用

优先用于下面这类任务：

- 看懂整个仓库、系统、方案或论文，并写成新人导览
- 重建 `what / how / why`
- 深挖单一模块、单一流程、单一风险或单一边界
- 做可行性判断、方案比较、证据分层
- 整理配置清单、接口清单、证据账本、矩阵、索引这类材料

默认不用于下面这类任务：

- 简单函数解释、单段代码说明、一次性问答
- 纯 bug 修复、纯实现建议、纯编码任务
- 不需要证据链、不需要结构化落盘的轻量回答

## 路由与更新

先判断这次要产出什么：

1. 如果这是第一次形成某个研究对象的实质性全局分析，产出主报告。
2. 如果这是单一模块、单一流程、单一风险、单一边界或单一对比维度，且需要独立证据链，产出 followup。
3. 如果这是证据整理、配置清单、接口清单、矩阵、索引、专项代码导读，产出 appendix。
4. 如果这轮只是补几条证据、补几句解释、补一张局部图或一个小 caveat，直接更新主报告，不新建文档。

最小示例：

- “帮我看懂整个仓库，写成新人导览” -> 主报告
- “单独深挖任务调度链路，从入口到错误处理拆开讲” -> followup
- “整理所有环境变量、默认值、来源和影响范围” -> appendix
- “把主流程里那个状态同步步骤再解释两句” -> 更新主报告

主报告是稳定入口。后续即使拆出 followup 或 appendix，主报告仍然要保持“先读这个就能建立全局”的作用。

优先拆成 followup，当且仅当：

1. 这个问题可以被清楚命名成一个独立专题。
2. 回答它需要相对独立的证据链、结构化展开或完整小结论。
3. 如果把它直接塞回主报告，会明显打断主线，或让主报告结构失衡。

优先拆成 appendix，当且仅当：

1. 用户要的是清单、矩阵、索引、证据账本或专项材料整理。
2. 主要价值是便于查阅、对照和枚举。叙事展开放到主报告或 followup。
3. 如果写回正文，会让材料细节淹没主线。

默认不要过度拆分。下面这些情况通常直接更新主报告：

- 用户只是追问一句补充解释
- 用户只要一个例子、一段额外证据、一个更清楚的表述
- 问题本质上仍在原章节的自然延长线上
- 新内容完全可以作为主报告中的小节或几段补充来承载

如果 followup 或 appendix 得出了会影响全局理解的新结论、边界、风险或设计解释，要把提炼版内容回灌到主报告对应位置。只回灌提炼后的结论、边界、风险、链接和一两句必要解释；不要整段复制。

## 三类产物的职责

### 主报告

- 这是稳定入口，负责总览、主链路、核心部件地图、项目结构设计、整体判断和继续阅读入口。
- 用户目标是理解项目、系统或架构全貌时，默认先产出主报告。
- 主报告必须单独解释项目结构设计；技术栈保留，但作为轻量辅助节，不要抢走主链路和结构设计的位置。

### followup

- 用于可以独立命名、独立阅读的单一专题。
- 它负责展开专题，不负责重新建立整个系统的全局入口。
- 默认不要复用主报告的整套总览骨架，除非用户明确要求把该专题重写成独立新人导览。

### appendix

- 用于查阅型材料，不负责重讲整套系统。
- 默认不要写成长篇叙事报告，不要重新讲完整主链路。
- 证据、配置、接口、模块映射等内容优先用表格、列表、矩阵或索引组织。

快速路径：

- 主报告 -> `references/report-template.md` + `references/report-sections.md`
- followup -> `references/followup-template.md`
- appendix -> `references/appendix-template.md`

详细章节模板不要在这里重复展开，需要时读取对应 `references/`：

- 主报告：`references/report-template.md` 和 `references/report-sections.md`
- followup：`references/followup-template.md`
- appendix：`references/appendix-template.md`

## 交付规则

- 先决定本轮交付强度，再决定是否落盘。
- 默认交付梯度从轻到重依次是：
  1. 只在聊天里回答
  2. 更新已有文档
  3. 新建单个主报告
  4. 新建 followup 或 appendix
- 默认先停在 1 或 2，不要一上来升到 3 或 4。
- 如果这是第一次让你整体介绍、看懂、导览当前项目，而用户没有明确禁止落盘，默认直接升到 3，新建主报告。
- 下面这些情况通常停在 1：
  - 用户明确说只想先聊清楚
  - 问题范围很窄，只需要简短结构化说明
  - 这轮主要是在确认方向，暂时不需要沉淀可复用文档
- 下面这些情况通常停在 2：
  - 已经存在同一研究对象的主报告、followup 或 appendix
  - 用户明确给出要更新的既有文件
  - 这轮只是补充证据、补几句解释、补一个小 caveat
- 只有当任务明显需要稳定文档资产、首次形成系统级导览，或用户明确要求“写成报告 / 落成文档 / 给新人看”时，才升到 3。
- 只有当子问题已经可以独立命名、需要独立证据链，或本轮目标本来就是清单 / 矩阵 / 索引这类材料归档时，才升到 4。
- 如果已经存在同一研究对象的文档集，后续追问优先更新它，避免内容只停留在聊天里。
- 默认优先更新已有文档；若用户没有指定路径，不主动创建新的目录体系、隐藏目录、索引页或归档结构。
- 若用户没有指定文件名，默认使用：
  - 主报告：`<project-name>-report.md`
  - followup：`<project-name>-<topic>-followup.md`
  - appendix：`<project-name>-<topic>-appendix.md`
- 如果项目名暂时无法可靠判断，主报告退回 `architecture-report.md`。
- 如果用户明确要求“只在聊天里回答”或明确给出目标文件路径，服从用户要求，不要擅自升级交付强度。

最终聊天回复应明确说明：

- 这次更新了主报告、followup、appendix，还是其中之一
- 相关文件路径或文件名
- 本轮关键结论是否已经回灌到主报告

## 文风

- 直接写事实、动作、判断和依据，少写铺垫性元话语。
- 避免 `不是……而是……`、`换言之`、`这意味着`、`为了方便理解`、`本文将`、`本文遵守` 这类套话。
- 少用 `可以看出`、`不难发现`、`值得注意的是` 这类空转判断句；能直接给结论就直接给。
- 需要比较方案、模块或角色时，直接列差异点、适用边界和证据来源，不靠修辞对比制造结论感。

## 图示与表达

- 只要主题涉及架构、运行链路、模块协作，就优先给图。
- Mermaid 优先选兼容性更高的简单写法，例如 `flowchart TD` 或 `flowchart LR`。
- 总览结构图的降级顺序：Mermaid 图 -> 分层列表或部件表 -> 极小的 ASCII 静态拓扑。
- 流程图的降级顺序：Mermaid 图 -> 编号流程。流程图默认不要降级成 ASCII。
- 一旦流程存在分支、回环、重试、异常路径或多角色协作，Mermaid 失败时必须改用编号流程。
- 编号流程每一步至少回答三件事：谁在动作、做了什么、产出了什么。
- 其余写作细则优先交给对应模板，不要重新堆回本文件。

## 何时读取 references

- 需要定义证据强度时，读取 `references/evidence-model.md`
- 需要系统评估可行性与风险时，读取 `references/feasibility-checklist.md`
- 需要输出新人友好的主报告时，读取 `references/report-template.md`
- 需要把主报告章节职责写得更稳定时，读取 `references/report-sections.md`
- 需要输出独立专题时，读取 `references/followup-template.md`
- 需要输出材料补充时，读取 `references/appendix-template.md`
- 需要参考本仓库示例时，读取 `references/senera-loop-example.md`

## 执行提醒

- 先用本地材料回答“代码里到底有什么”，再用外部材料回答“这种设计在公共世界里站不站得住”。
- 外部来源除了说明“支持了什么”，还要说明“没有证明什么”。
- 当用户要的是新人可读性，先讲“结构怎么组织、系统怎么跑”，再讲“系统为什么这样设计”。
